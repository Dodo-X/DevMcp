# DevPartner 前端界面优化方案与调整结果

> 重构对象：`backend/api_gateway/dashboard.html`（被 `/dashboard` 路由直接读出的单文件运维面板）
> 设计角色：UI Designer —— 以设计系统、组件库、无障碍与像素级一致为目标
> 验证：JS `node --check` 通过；JS 引用的 115 个元素 ID 与 HTML 全部对应（NONE missing）；本地启动服务 `/dashboard` 返回 ~96KB，`{{VERSION}}` 已正确替换。

---

## 一、本次直接修改的内容（四个维度）

### 1. 用户需求（以任务完成为中心）
- **重排信息架构**：原 7 个扁平 Tab（总览把用户侧与运维侧混在一起）改为**侧边栏分组导航**——「工作区」（概览 / 成长 / 知识库 / 项目 / 报告）与「管理」（优化建议 / 系统运维），让开发者最常用的成长、报告、知识库前置，深层运维下沉。
- **概览页减负**：从「什么都有」改为「一眼看懂」——仅保留 4 个核心 KPI + 会话趋势 + 系统脉冲（LLM/内存）+ 队列概览 + 快捷操作（生成今日日报 / 清算待分析），并显著提示待清算数据。
- **快捷入口**：概览页一键「生成今日日报」「清算待分析数据」，减少操作步骤。

### 2. 界面美观（统一的视觉语言）
- 建立 **Design Token 体系**（颜色 / 间距 / 圆角 / 阴影 / 缓动），杜绝原版散落的硬编码色值。
- 克制的「技术控制台」暗色风：单一靛蓝主色（`--accent`）+ 语义色（绿/琥珀/红），避免泛 AI 审美的霓虹发光与紫蓝渐变。
- 数字采用 `tabular-nums` 等宽对齐、负字距，呈现「仪表」质感；卡片统一圆角/边框/悬浮抬升；标题用小竖条 accent-bar 强化层级。

### 3. 功能划分（信息架构清晰）
- 用户侧（看成长、读报告、搜知识）与管理侧（审建议、管系统）物理分离，避免功能堆砌。
- 同类聚合：运维页把「诊断 / 完整性 / LLM / 拦截器 / 清理 / 归档 / Pending / 处理器·回调」分卡片归组；原「概览」里 5 张运维大卡全部迁走。

### 4. 用户交互（反馈 / 动效 / 表单 / 快捷键 / 无障碍）
- **加载态**：原「加载中…」全部替换为**骨架屏（shimmer）**。
- **操作反馈**：原 `alert()` 改为**右上角 Toast**（成功/错误/警告/信息四态），破坏性操作（归档、清理）改为**确认弹窗**。
- **动效**：页面/卡片切换统一 `cubic-bezier(0.16,1,0.3,1)` 缓动，并尊重 `prefers-reduced-motion`。
- **表单体验**：日期/周/月/年选择器、范围下拉、聚焦描边、加载禁用态。
- **快捷键**：`1–7` 切页、`R` 刷新、`/` 聚焦知识库搜索、`?` 帮助、`Esc` 关闭弹窗。
- **无障碍**：`跳转主内容` 链接、`<nav>/<main>` 语义结构、`aria-current`、`:focus-visible` 描边、弹窗 `role="dialog"`、`aria-live` 告知区。
- **安全**：报告预览由「内联 HTML 拼接」改为**安全 Markdown 渲染**（转义后仅渲染有限标签），降低 XSS 风险。

## 二、建议移除 / 下沉的前端功能（当前「不需要」放前台）

| # | 功能 | 处置建议 | 理由 |
|---|------|----------|------|
| 1 | **请求拦截器**（拦截 Ollama 请求日志） | 下沉为「开发者工具」，前端默认隐藏 | 低频调试用、占内存，普通用户无感 |
| 2 | **数据清理 / 数据归档 / Pending 任务清理** | 移入独立「运维后台」或仅保留 API，前端默认折叠 | 高危写操作，误触风险高，不应与日常面板并列 |
| 3 | **已注册处理器 / 回调统计明细** | 合并为单张「系统组件」卡片 | 与诊断信息重复，拆成两张卡是功能堆砌 |
| 4 | **概览页的运维大杂烩**（任务运维指标/并发诊断/Handler/回调/健康） | 已全部迁出到「系统运维」页 | 概览应为用户视角的「一眼看懂」，不是引擎仪表盘 |
| 5 | **纯文本 `ops-result` 控制台输出** | 改为结构化卡片/表格 | 原始 JSON 堆叠可读性差 |
| 6 | **报告内联 HTML 渲染** | 已改为安全 Markdown 渲染 | 防 XSS，统一排版 |

> 结论：前端的「管理 / 运维」部分应默认只暴露**只读监控 + 审核流**（优化建议），写操作与调试器收口到独立后台或 CLI/接口。

## 三、缺失的功能（前端应该有，但现在没有）

> **产品决策**：系统**不需要登录 / 多用户鉴权**——服务启动后 `/dashboard` 直接可访问，MCP 工具也能正常调用。因此鉴权相关能力**不在本次范围**，以下条目均不含登录。

1. **会话历史浏览**：目前仅有单条状态查询，缺列表与详情视图。
2. **用户画像 / 开发者能力模型**视图：后端有画像融合逻辑，但前端无对应页面。
3. **报告内搜索 / 跨周期对比**：报告只能按类型列清单，无法检索内容或对比两期。
4. **趋势图时间范围可配置**：当前硬编码近 30 天，前端无法选。
5. **任务实时进度**：已有进度接口但未在前端使用，仍靠轮询。
6. **设置页**：模型名、端口、主题、自动刷新等无法在界面调整。
7. **通知中心 / 待办**：pending analyses 仅做了概览提示，无统一通知聚合。
8. **知识图谱可视化**：知识提取已沉淀图谱数据，前端无可视化。
9. **浅色主题切换**：当前仅暗色。
10. **报告下载**：Markdown 文件已在磁盘生成，但前端无「下载」入口。

## 四、需要新增 / 调整的前端接口

| 方法 | 路径 | 用途 | 说明 |
|------|------|------|------|
| `GET` | `/api/conversations` | 浏览对话历史（分页 + 筛选） | 补齐会话历史浏览 |
| `GET` | `/api/conversations/{id}` | 对话详情（步骤、分析、总结） | 与列表配合 |
| `GET` | `/api/profile` | 当前开发者画像与能力模型 | 补齐用户画像视图 |
| `GET` | `/api/reports/search?q=` | 跨报告全文搜索 | 补齐报告内搜索 |
| `GET` | `/api/trends/system?days=7\|30\|90` | 支持时间范围参数 | 调整现有接口（原硬编码 30 天） |
| `GET` | `/api/tasks/progress/stream`（SSE/WS） | 任务实时进度推送 | 替代 30s 轮询 |
| `GET` | `/api/notifications` | 通知 / 待办聚合（pending analyses、失败任务等） | 补齐通知中心 |
| `GET` | `/api/knowledge/graph` | 知识图谱数据（节点 + 边） | 供可视化 |
| `GET/POST` | `/api/settings` | 设置读取与写入 | 补齐设置页、主题、自动刷新开关 |
| `GET` | `/api/export/report?type=&name=` | 报告文件下载 | 补齐报告下载 |
| `PATCH` | `/api/growth/{id}` | 完整更新建议（标题/状态/意见） | 调整：当前仅有 `review`/`apply` 动作端点 |

> 注：上表新增接口均应以「统一返回体 `{code,message,data}`」封装，并复用现有 `foundation/api_response` 的 `ok/fail/page` 工厂，保持前后端契约一致。

## 五、建议的最终前端信息架构

```
工作区（开发者日常）
├─ 概览       KPI / 趋势 / 系统脉冲 / 队列概览 / 快捷操作
├─ 成长       技能趋势 / 领域清单 / 效率 / 热力图 / 时间线
├─ 知识库     AI 匹配 / 精确搜索 / 浏览
├─ 项目       对接项目 / 项目知识检索
└─ 报告       生成 / 列表 / 预览（含搜索、对比、下载）

管理（运维 / 审核）
├─ 优化建议   审核 / 应用工作流
├─ 系统运维   诊断 / 完整性 / LLM / 进度（只读为主）
└─ 设置       模型 / 主题 / 自动刷新（需新增 /api/settings）
```
